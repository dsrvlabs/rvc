# rvc v0.2.0 â€” Phase 1 Audit Remediation Release

**Release date:** TBD
**Target tag:** `v0.2.0` on the same commit for `bin/rvc` and `bin/rvc-signer`.

This release closes all 4 Critical findings (C-1..C-4) from the audit, plus
Phase 0 dependency hygiene. It introduces **proto v2** as the only signing
contract, embeds slashing protection inside `rvc-signer` with per-`client_cn`
scoping, and enforces WAL durability on the slashing database.

## Hard breaks for operators

### Proto v2 â€” typed RPCs only (BREAKING)

The legacy `signer.proto` raw-root RPC is replaced by `signer.v2.proto`
(typed RPCs). There is **no `--legacy-proto-v1` shim**. Both `bin/rvc` and
`bin/rvc-signer` must be upgraded together. `bin/rvc` logs the contract
version at startup and refuses to talk to a v1 signer.

#### Typed-RPC migration table

| Legacy call | v2 RPC | Notes |
|---|---|---|
| `Signer.Sign(signing_root, pubkey)` for blocks | `SignerService.SignBeaconBlock(SignBeaconBlockRequest)` | request carries `pubkey`, `fork_info`, `block_ssz`, `fork_id` |
| `Signer.Sign(signing_root, pubkey)` for blinded blocks | `SignerService.SignBlindedBeaconBlock(...)` | same shape with `blinded_block_ssz` |
| `Signer.Sign(signing_root, pubkey)` for attestations | `SignerService.SignAttestationData(...)` | explicit `AttestationData` (slot, index, source/target Checkpoints, beacon_block_root) |
| `Signer.Sign(signing_root, pubkey)` for aggregates | `SignerService.SignAggregateAndProof(...)` | `aggregate_ssz` (full `AggregateAndProof`), `aggregator_index`, `selection_proof` |
| `Signer.Sign(signing_root, pubkey)` for sync committee messages | `SignerService.SignSyncCommitteeMessage(...)` | `slot`, `beacon_block_root` |
| `Signer.Sign(signing_root, pubkey)` for sync aggregator selection | `SignerService.SignSyncAggregatorSelectionData(...)` | `slot`, `subcommittee_index` |
| `Signer.Sign(signing_root, pubkey)` for contributions | `SignerService.SignContributionAndProof(...)` | `contribution_ssz`, `aggregator_index`, `selection_proof` |
| `Signer.Sign(signing_root, pubkey)` for builder registrations | `SignerService.SignBuilderRegistration(...)` | server uses fixed `GENESIS_FORK_VERSION` + `ZERO_HASH` per MEV-Boost spec |
| `Signer.Sign(signing_root, pubkey)` for randao | `SignerService.SignRandaoReveal(...)` | `epoch` |
| `Signer.Sign(signing_root, pubkey)` for voluntary exits | `SignerService.SignVoluntaryExit(...)` | `epoch`, `validator_index`. EIP-7044 cap is the **caller's** responsibility â€” see `crypto::capella_capped_fork_version` |

DVT operators get three `PeerSignerService` typed RPCs:
`PartialSignBeaconBlock`, `PartialSignAttestationData`,
`PartialSignSyncCommittee`. Each request carries `requester_index` (field 1)
which is enforced against the peer's CN via `dvt-allowed-peers.toml`.

### `rvc-signer` embeds slashing protection (BREAKING for operators with no DB)

`rvc-signer` now refuses to start without a slashing database. Default DB
path: `<data_dir>/signer-slashing.db`.

To opt out (NOT recommended), set BOTH:

```sh
rvc-signer --disable-slashing-protection
RVC_ALLOW_INSECURE=true ./rvc-signer ...
```

Both flags are required. Setting only one causes startup to refuse with an
actionable error.

### Slashing DB schema v2 (auto-migration on first open)

Existing v1 SlashingDb files are migrated **eagerly** on first open by the
new binary. Before any ALTER, the original DB is backed up to
`<path>.bak.<UNIX_TS>` (atomic copy + `sync_all`). The migration is
idempotent â€” re-opening a v2 DB takes no backup and no ALTER.

Operators should:
- Verify the `.bak.<TS>` file appears next to your `slashing.db` after first
  start.
- Optionally prune old backups: `find <data_dir> -name 'slashing.db.bak.*' -mtime +30 -delete`.

The `slashing.db.bak.*` files are operator-managed; rs-vc does not auto-prune.

### WAL hard-fail (BREAKING for non-WAL filesystems)

`rvc-signer` refuses to open the slashing DB if SQLite returns a non-WAL
journal mode. Operators on tmpfs / NFSv3 / SMB may opt out:

```sh
RVC_ALLOW_NON_WAL_SLASHING_DB=true ./rvc-signer ...
```

This logs a loud `error!` warning ("running without WAL â€” slashing
protection durability degraded") and continues. Use only as a temporary
workaround â€” durability across host crashes is degraded.

### DVT operators: new `dvt-allowed-peers.toml`

DVT-enabled rvc-signer instances require a `dvt-allowed-peers.toml` file:

```toml
[[peer]]
peer_cn = "peer-A"
share_index = 1

[[peer]]
peer_cn = "peer-B"
share_index = 2
```

The path is set via `--dvt-allowed-peers <PATH>`. Each request's
`requester_index` field is enforced to match the peer's `share_index`. Peers
with valid certs but missing from the allow-list are rejected with
`UNAUTHENTICATED`.

## Findings closed

### Critical (C-1..C-4)

- **C-1** SSZ `BlockContents` parser: body region is now bounded by the

  implements `RawSigner`. It implements `TypedSigner` only. The server
  reconstructs the signing root from typed inputs and stages the slashing

## Architectural notes

### Stage â†’ sign â†’ commit (A15)

Slashable signing now uses an RAII `StagedBlock<'_>` / `StagedAttestation<'_>`
guard from `crates/slashing`. The guard opens an immediate transaction,
runs the EIP-3076 violation check, and stages the INSERT. The transaction
is held across the signer call. On signer success â†’ `staged.commit()`. On
signer error or guard drop â†’ `ROLLBACK`. **No phantom rows on signer
failure.**

Server-side dispatch in `bin/rvc-signer` runs the entire stage â†’ sign â†’
commit triple inside `tokio::task::spawn_blocking` (the staged guard holds
a `parking_lot::MutexGuard` which is `!Send`). The async `backend.sign`
call is driven via `Handle::current().block_on(...)` from inside the
blocking closure.

### Per-`client_cn` scoping (A1)

Every slashing-DB row now carries a `client_cn TEXT NOT NULL DEFAULT
'__legacy__'` column. Two sentinels:

- `'__legacy__'` â€” only the column DEFAULT for pre-migration v1 rows.
- `'local-vc'` â€” VC-side runtime writes (set explicitly by `crates/signer`).

DVT peer writes use the peer's CN as the `client_cn`. Uniqueness indexes
are `(client_cn, pubkey, target_epoch)` and `(client_cn, pubkey, slot)` so
each tenant has an independent watermark namespace.

### Per-row `genesis_validators_root` column

Schema v2 adds a nullable `genesis_validators_root` column to attestations
and blocks. The column is **populated but not yet enforced** â€” enforcement
against the metadata-pinned value lands in the next release as M-6
(ISSUE-3.5). Do not rely on this column for forensics until M-6 ships.

## Known limitations / follow-ups

- **gRPC remote signers are runtime-broken in this release.** The
  CompositeSigner boundary correctly refuses raw-root signing for gRPC
  remote pubkeys (closes C-2/C-3), but the VC-side `SignerService` call
  sites in `crates/signer/src/lib.rs` have not yet been migrated to
  dispatch typed methods to the gRPC remote. Operators using gRPC remote
  signers (i.e., the legacy v1 client config) will see signing errors at
  runtime. Migrate to LocalSigner or wait for the follow-up release that
  completes the call-site migration. The C-2/C-3 hardening is in place â€”
  the issue is functional, not security.
- **`proto/signer.proto` is still in the tree** (tonic-build still
  compiles it for `ListPublicKeys` used by `GrpcRemoteSigner::connect`).
  Full removal lands when `connect` flips to v2 `ListPublicKeys`.
- **`x509-parser` swap (M-4) deferred to next release.** The current
  hand-rolled DER scanner in `audit/cn.rs` returns the *last* CN match per
  RDN; a crafted cert with two CNs can spoof the audit-log CN.
