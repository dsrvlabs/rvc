# rvc v0.3.0 — Mainnet GA Release

**Release date:** 2026-05-02
**Target tag:** `v0.3.0` on the same commit for `bin/rvc` and `bin/rvc-signer`.

This release closes all 12 Medium audit findings (M-1..M-12), flips the
`InsecureGate` default from `Warn` to `Refuse`, and declares Mainnet GA status.
Both binaries must be upgraded together.

---

## Breaking changes for operators

### InsecureGate default flipped: `Warn` → `Refuse` (ISSUE-3.13 / NFR-10)

**Action required for any operator using `--insecure` or an `http://` remote signer.**

`InsecureMode` defaults to `Refuse` in this release.  Operators who relied on
the Phase-2 warn-only behaviour (the binary logged an error but continued) will
now see a **hard startup failure**.

#### H-9: `rvc-signer --insecure` (bind-address gate)

To use `--insecure` you must now satisfy BOTH:

1. Set `RVC_SIGNER_ALLOW_INSECURE=true` in the environment.
2. Bind to a loopback address (e.g. `127.0.0.1` or `::1`).

If either condition is not met, `rvc-signer` refuses to start with an
actionable error that names the env var.  The recommended fix is to switch to
TLS (`--tls-cert`, `--tls-key`, `--tls-ca-cert`).

#### H-10: `GrpcRemoteSigner` / `RemoteSigner` plaintext URLs

`http://` remote-signer URLs now hard-fail unless
`RVC_REMOTE_SIGNER_ALLOW_INSECURE=true` is set.  Switch to `https://` or set
the env var (only for isolated lab environments).

> **Note:** `L-10` (metrics bind gate) is scoped for Phase 4 and is **not**
> wired in this release.

### Post-import doppelganger window (ISSUE-3.11 / M-12)

Newly imported keystores no longer become attesting-enabled immediately.  They
enter the configured doppelganger window first.  Operators who relied on instant
activation after `POST /eth/v1/keystores` must account for the window delay
(default: `doppelganger_epochs` from config).

Use the key status endpoint to observe `"imported, in doppelganger window"` vs
`"imported, attesting"`.

### Backup-file retention (Phase 1 ISSUE-1.9 — still operator-managed)

Schema-v2 migration (introduced in v0.2.0) creates `.bak.<UNIX_TS>` backup
files next to `slashing.db` on first open.  These are **not** auto-pruned.
Recommended cleanup snippet:

```sh
find <data_dir> -name 'slashing.db.bak.*' -mtime +30 -delete
```

## Medium findings closed (M-1..M-12)

### M-1 — SignerService stage + commit on success (ISSUE-3.1)

`SignerService::sign_attestation` and `sign_block` now use `stage_*` +
`commit()` instead of `check_and_record_*`.  On signer failure, `discard()`
rolls back the staging transaction — no phantom rows.  A subsequent retry with
a different signing root succeeds without spurious DoubleVote rejection.

### M-2 — Local AttestationData sanity check (ISSUE-3.2)

`validate_attestation_data()` runs before every sign: verifies
`target.epoch == data.slot / SLOTS_PER_EPOCH`, `source.epoch <= target.epoch`,
and `data.slot` within ±2 slots of the local clock.  Confused/malicious BN
data is rejected before reaching the signer.

### M-3 — Circuit-breaker atomic ordering (ISSUE-3.3)

`crates/builder/src/circuit_breaker.rs` replaced `Ordering::Relaxed` with
`AcqRel` / `Acquire` / `Release` ordering.  Stale breaker-state reads on ARM
are prevented.

### M-4 — CN extraction via x509-parser (ISSUE-3.4)

The hand-rolled DER scanner in `bin/rvc-signer/src/audit/cn.rs` that returned
the *last* CN OID (spoofable by a crafted cert with two CN RDN entries) is
replaced with `x509-parser`-based parsing that returns the *first* CN per RDN
rules.

### M-5 — TruncatedPubkey in audit logs (ISSUE-3.4)

`bin/rvc-signer` audit-log entries now show an 8-character prefix + `…` instead
of the full 96-character pubkey hex.  Implemented via `crypto::logging::TruncatedPubkey`.

### M-6 — Per-row genesis_validators_root re-check (ISSUE-3.5)

Every `check_and_record_*` and `stage_*` call now validates the caller-supplied
`gvr` against the metadata-pinned value in the slashing DB.  A chain-swap
(operator accidentally pointing at a different chain's BN) returns
`SlashingError::GenesisRootMismatch { expected, got }`.

### M-7 — Doppelganger BN-derived clock (ISSUE-3.6)

`DoppelgangerService` derives `current_epoch` from `genesis_time` + monotonic
elapsed (`Instant::now()`) rather than from the wall-clock.  NTP jumps or
clock manipulation no longer shift the doppelganger window mid-run.

### M-8 — Keymanager-API error sanitization (ISSUE-3.7)

All three error-leaking spots in `crates/keymanager-api/src/handlers.rs`
(lines 90-94, 122-126, 304-306) now wrap internal errors in
`ApiError::Internal("internal error")`.  Full error chains are logged at
`error!` level with a request ID; clients receive only a stable generic message.

### M-9 — set_attesting_enabled confirmation + rate-limit (ISSUE-3.8)

`set_attesting_enabled` is rate-limited to 1 call per 60 seconds.  A second
call within the window returns HTTP 429 with a `Retry-After` header.  Every
call emits an audit log entry with caller identity and timestamp.

### M-10 — Tonic server limits (ISSUE-3.9)

`bin/rvc-signer` now applies hardened Tonic limits via
`tls::server_builder::hardened_server_builder()`:
- `concurrency_limit_per_connection(32)`
- `max_concurrent_streams(Some(64))`
- `timeout(Duration::from_secs(10))`
- Per-service `max_decoding_message_size(1 << 20)` (1 MiB)

### M-11 — Attestation timing rounding (ISSUE-3.10)

`slot_duration / 3` integer division is replaced by
`Duration::from_millis(slot_duration_ms / 3)`.  Attestation fires at exactly
`slot_start + slot_duration / 3` on non-12 s slots (e.g., 6 s testnet slots
now fire at exactly 2.000 s, not up to 1 s early).

### M-12 — Post-import doppelganger window (ISSUE-3.11)

See "Breaking changes" above.  Newly imported keys run the configured
doppelganger window before becoming attesting-enabled.

## Risk mitigations

### tx_hold_duration_ms histogram (ISSUE-3.12)

A `rvc_signer_slashing_tx_hold_duration_ms` Prometheus histogram (labeled
`kind=block|attestation`) records the duration each `stage_*` SQLite
transaction is held open across the signer call.  Operators can alert if
p99 > their threshold.

## Upgrade path from v0.2.0

1. Stop `rvc` and `rvc-signer`.
2. Replace both binaries from the same `v0.3.0` build.
3. If using `--insecure`: ensure `RVC_SIGNER_ALLOW_INSECURE=true` is set and
   the bind address is loopback, OR switch to TLS.
4. If using `http://` remote signers: set `RVC_REMOTE_SIGNER_ALLOW_INSECURE=true`
   or switch to `https://`.
5. Expect a doppelganger window for any keys imported after startup (M-12
   behaviour change).
6. Restart `rvc-signer` then `rvc`.
