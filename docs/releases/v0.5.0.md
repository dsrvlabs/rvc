# rvc v0.5.0 — Audit-remediation dev-run, intra-slot BPS timing, and spec-conformance hardening

**Release date:** 2026-06-20
**Target tag:** `v0.5.0` on the same commit for `bin/rvc`, `bin/rvc-signer`, and `bin/rvc-keygen`.

This release lands a thorough, test-first re-execution of the audit-remediation
program ("dev-run") covering the Critical/High/Medium finding classes, converts
all intra-slot duty timing to a basis-points (`due_ms`) model in the
`rvc-timing` crate, fixes four duty-signing spec divergences validated against
the consensus spec / Lighthouse, and adds an optional wildcard keystore-password
default. All three binaries must be upgraded together.

> **Scope.** 95 commits on `develop` since `v0.4.0` (`v0.4.0..develop`). This is
> a normal `v0.4.0 → v0.5.0` upgrade: `develop` is built directly on top of the
> `v0.4.0` release commit (the workspace was at `0.4.0` going in), so everything
> below is what is new relative to a `v0.4.0`-class build. If you are upgrading
> from a `v0.2.0`/`v0.3.0`-class build, also read `docs/releases/v0.3.0.md` and
> `docs/releases/v0.4.0.md` and apply their operator actions.

## Breaking changes for operators

There are **no new required-config or hard-fail breaking changes in this
release's commit range.** The InsecureGate `Refuse` default, the metrics-bind
refusal, and the keystore-hot-reload opt-in were all introduced in v0.3.0/v0.4.0
and are unchanged here. The items below are behavioral changes worth knowing
before you upgrade.

### Slashing DB auto-migrates to schema v3 on first open (`c4b17d4`)

On first open after upgrade, the slashing-protection database **automatically
migrates from schema v2 to v3** (re-keying slashing-check indices from
`client_cn`-scoped to **pubkey-scoped**, per finding DVT-1/CN-1). The migration
runs inside a single `BEGIN IMMEDIATE … COMMIT` block (any failure rolls back —
no partial state) and is idempotent (`schema_version >= 3` is a no-op). A
timestamped backup `<db-path>.bak.<UNIX_TS>` is written before any `ALTER` fires
(`db.rs:450`).

- The migration back-fills any NULL `genesis_validators_root` rows from the
  pinned metadata GVR and **fails closed** if data rows exist but no GVR is
  pinned (`migration.rs:158`). Ensure the DB has its GVR pinned before upgrading.
- Duplicate row-pairs that were only distinguishable by `client_cn` under v2 are
  collapsed safely. `client_cn` is retained as an **audit-only** column.

**Action:** none required for normal deployments — the migration runs on
startup. Confirm the process can write the backup next to the DB, and keep that
`.bak` until you have verified the upgraded client signs correctly. The migration
is one-way; do not downgrade `rvc-signer` against a v3 DB.

### Unknown pubkeys now fail closed in the signing-enablement pre-filter (`392da00`)

`ValidatorStore::is_attesting_enabled` was renamed to `is_signing_enabled` and
its unknown-pubkey default flipped from **fail-open (`true`) to fail-closed
(`false`)**: a pubkey the store has never tracked is never permitted to sign. To
prevent a liveness regression, startup now registers every loaded keystore
pubkey into the store (`register_loaded_validators`, called from both
`ServiceBuilder::build_services` and `bin/rvc` startup), so the tracked set
equals the loaded-key set. A key already disabled by the doppelganger window or
by `--validators-config` TOML is left disabled.

**Action:** none for standard keystore-loaded deployments. If you drive
enablement through a custom `ValidatorStore` population path, ensure every pubkey
you expect to sign is registered, or it will be silently skipped (with a `warn!`).

### Intra-slot duty timing shifts by milliseconds (`21a24ac`, `dee47e6`, `d67e3db`, `68608df`)

Duty firing moved from whole-second integer fractions to a basis-points formula
`due_ms(bps, slot_duration_ms) = bps * slot_duration_ms / BASIS_POINTS` (floor).
These are intentional value corrections, not regressions:

- Mainnet 1/3 attestation trigger: **4000 ms → 3999 ms** (`3333 * 12000 / 10000`).
- Mainnet 2/3 aggregation trigger: **8000 ms** (unchanged).
- 6 s networks: 1/3 trigger 2000 ms → **1999 ms**; 7 s networks: stays **2333 ms**.
- `attestation_time` (seconds API): genesis+4 → **genesis+3** (`timer.rs:292`).

**Action:** none. If you have external monitoring that asserts the exact
sub-second offset at which attestations/aggregations fire, update those
thresholds.

### Optional wildcard keystore-password default (`c0ea2d9`, `cddf867`, `e6edabd`)

The keystore password file now accepts a single `*=<password>` line that sets a
default password for every keystore lacking its own `pubkey=password` entry.
Precedence is **exact pubkey → `*` wildcard → skip** (the skip path emits the
existing `No password found for public key …` warning). This is fully backward
compatible — files without a `*` line behave exactly as before; the wildcard is
purely additive.

> **Security note (shared-secret blast radius):** the `*` password decrypts
> *every* un-overridden keystore, so anyone who can read the password file
> effectively holds all those keys. If you adopt this, scope the file tightly
> and keep `chmod 600` on `passwords.txt`. See `docs/running-guide.md` and
> `config.example.toml`.

## Audit-remediation dev-run

A clean, RED→GREEN→REFACTOR re-execution of the remediation program (Issue
1.0–2.13), re-doing the Critical/High/Medium finding classes thoroughly.
Foundations first, then wiring, each with a reproduction test and a follow-up
review pass.

### Foundations and standing gates (Issues 1.0–1.7)

- **Canonical type seams** (`53e9752`): `eth-types::canonical` adds `PubkeyHex` /
  `GvrHex` / `SigningRootHex` with strict hex parsers.
- **InsecureGate primitive** (`b1b8a89`): the `eth-types::insecure::InsecureGate`
  decision primitive used by the plaintext-URL and non-loopback refusals.
- **Signing-enablement seams** (`ff661a9`): `SigningEnablement` +
  `FailClosedDefault` trait seams (`#[must_use]`, dyn-compatible).
- **Read-only slashing seam** (`09e6e41`): `SlashingDbReader`, hardened to **fail
  closed** when the DB has no pinned GVR or a mismatched GVR (Issue 1.4 SEC-001)
  — it returns the last-signed record only on an exact GVR match, so a
  doppelganger safe-skip can never be unlocked by foreign chain history.
- **Architecture/registry gates** (`6ba3c4e`): a standing `architecture_no_cycles`
  CI gate, and a dev-only `signer-registry` crate that enumerates the methods
  served on the live listener (feeds the M4 enumeration gate, below).

### High-class wiring (SS-1, KM-1, DVT-1/CN-1)

- **SS-1 — v1 raw-root sign service removed from the live listener** (`0b157cf`):
  `rvc-signer` no longer serves the v1 raw-root `sign` service on the live
  listener; only the v2 typed-RPC service is served, and the connect-time
  `ListPublicKeys` probe was switched to the v2 client (`6ba3c4e`). This closes
  the v1 raw-root bypass. A standing strict enumeration gate (M4) fails the build
  if a non-allow-listed method is exposed on the live listener.
- **KM-1 — atomic interchange export on delete** (`399485c`, `125aec1`): `DELETE`
  on the keymanager API now takes a true atomic interchange snapshot and **fails
  closed** (aborts the deletion) if the export fails or is incomplete, so a key
  is never removed without a verified slashing-history export.
- **DVT-1/CN-1 — pubkey-scoped slashing** (`c4b17d4`): slashing checks are keyed
  on `pubkey` (+ GVR), not `client_cn`; `client_cn` is dropped from the `stage_*`
  signatures and retained audit-only via `audit_log`. The dead `ScopedSlashingDb`
  was removed. This closes the cross-CN double-sign acceptance reproduced by the
  RED test, and ships the schema v2→v3 migration described above.

### Doppelganger protection (D-1, D-2, S-3)

- **D-1 — forward-window state machine** (`0486585`): `ForwardWindowMachine`
  withholds signing for the full `monitoring_epochs` window and is restart-aware
  (safe-skip only when the complete window has been observed).
- **D-2 — fail closed on missing liveness** (`2dc2641`): incomplete/missing
  liveness data no longer lets a validator reach `Safe`; `Safe` requires complete
  window observation.
- **S-3 — detection always invoked** (`2bdf929`): the `if current_epoch > 0`
  startup guard was removed so doppelganger detection always runs; epoch 0 is an
  **explicit pre-genesis bypass** that logs the decision (no silent skip).

### SigningGate end-to-end wiring (D-3)

- **Gate core** (`0dd44bc`, `cae7518`): `SigningGate` + `ValidatorLockMap` —
  per-validator locking, fail-closed default, slashable and non-slashable paths,
  with the aggregate chain-of-custody skeleton.
- **Routed through the signer**: every `rvc-signer` typed handler routes through
  the gate; SS-2/SS-3 aggregate paths no longer stage to the slashing DB.
- **Orchestrator gates every path per-pubkey** (`7b73b03`): block proposal, sync
  messages, sync contributions, and aggregation/selection-proof all check
  enablement per-pubkey; disabled validators are skipped with a `warn!`. A
  grep-gate prevents any direct composite-signing call outside `crates/signer`.
- **Fail-closed enablement** (`392da00`): the `is_signing_enabled` rename/flip
  described above, plus the orchestrator attestation gate failing **closed** on
  an unresolved or non-decoding pubkey (e.g. a `0X`-prefixed input).

### Concurrency and verification (KM-2, M4/M6)

- **KM-2 — keymanager concurrent delete/re-import race** (`f94c0bd`): closed via
  cancel-on-insert with a single lock and self-pruning; the review pass closed a
  residual enable-path TOCTOU and a delete-path token leak.
- **M4/M6 verification suite** (`fe1aee8`): the PRD M4 (live-listener method
  enumeration) gate flipped to **strict** (only v2 typed methods may be live),
  and the M6 per-row GVR re-check verified end-to-end.

## Duty-signing spec conformance (vs consensus spec / Lighthouse)

Four duty-domain divergences were found and fixed; golden roots were re-derived
from an independent consensus-spec SSZ oracle.

- **Electra attestation leaf order** (`2efac13`): `ElectraAttestation`
  tree-hashing emitted the inner SSZ leaves swapped (`committee_bits` and
  `signature`), so every Electra-era `SignedAggregateAndProof` was signed over
  the wrong root and rejected on the `beacon_aggregate_and_proof` gossip topic.
  Corrected to strict field order; the signed-aggregate root is now byte-equal to
  the spec value `0x26b23c…914f`, pinned by a byte-literal KAT.
- **Builder/MEV domain fork version** (`0cbf8af`, `52e6b31`, `4eae10d`): the
  standalone signer hardcoded `genesis_fork_version = [0u8;4]`, so
  builder-registration signatures failed to verify on non-mainnet networks
  (Holesky/Sepolia/Hoodi). The domain is now computed from the request's fork
  version (empty ⇒ mainnet `[0u8;4]` for back-compat; exactly 4 bytes ⇒ the
  network version; any other length ⇒ `InvalidArgument`), and `grpc-signer` sends
  the builder genesis fork version on the wire. Mainnet output is unchanged
  (`0x00000000`); a Holesky KAT (`0x01017000`) pins the non-mainnet path.
- **Bitlist tree-hash limit-padding** (`b1475af`): `bitlist_tree_hash_root`
  merkleized only the populated data chunks and omitted the `Bitlist[N]`
  chunk-count limit, so the `aggregation_bits` leaf — and the whole attestation
  and signed-aggregate root — was not byte-equal to spec and spec-compliant peers
  rejected the gossip. The limit is now threaded through with the correct
  `mix_in_length`.
- **BPS slot timing** (`68608df`): the intra-slot timing correction, covered
  above.

### Security hardening caught in review (`35cffee`)

The bitlist limit-padding fix (`b1475af`) carried a **liveness-DoS regression**
that was caught and fixed before release. With the merkle tree sized to a fixed
chunk-count, an attacker-influenced, length-unvalidated `aggregation_bits` from
the beacon node that exceeded the `Bitlist[N]` byte limit overflowed the tree:
`MerkleHasher` returned `MaximumLeavesExceeded`, which was previously `.expect()`-ed
and **panicked the signing path** before the orchestrator's skip-guard could
catch it. The fix length-validates against the byte limit (`N/8 + 1`) up front
and propagates the `MerkleHasher` error via `?`, so an over-length bitlist is
rejected and skipped, not panicked. The largest valid bitlist (all `N` bits set)
fits exactly and still hashes.

## Other changes

- **Startup logs restored** (`e8dad72`): pad the empty boxed-layers `Vec` with
  `Identity` so startup logging emits again when no optional tracing layers are
  configured.
- **Test-suite speed-up ~16x** (`9d42e47`): optimize dependency build profiles
  and drop the in-test `cargo build`, dramatically shortening a clean workspace
  test run (now ~24 s).
- **Byte-pinned duty-signing KATs** (`bf3607b`, `6805fd1`, `d67e3db`):
  known-answer vectors across the crypto/sync/timing duty paths (including Gloas
  `due_ms` vectors), so the corrected roots and offsets cannot silently drift.
- **Docs/cleanup** (`ed6896b`): removed deprecated plan/remediation tracking docs
  after the dev-run completed (use `git log`).

## Upgrade path from v0.4.0

1. Stop `rvc`, `rvc-signer`, and any `rvc-keygen` jobs.
2. Replace all three binaries from the same `v0.5.0` build — they must be
   upgraded together.
3. **Slashing DB:** ensure the slashing-protection DB has its
   `genesis_validators_root` pinned, and that the process can write a backup next
   to the DB file. On first start the DB **auto-migrates v2→v3** and writes
   `<db-path>.bak.<UNIX_TS>` before altering. The migration fails closed if data
   rows exist without a pinned GVR. Keep the `.bak` until the upgraded client is
   verified signing.
4. **Signing enablement:** standard keystore-loaded deployments need no action —
   loaded pubkeys are auto-registered at startup. If you populate a custom
   `ValidatorStore`, register every pubkey that should sign, or it will be
   skipped (unknown pubkeys now fail closed).
5. **Wildcard password (optional):** to use one shared keystore password, add a
   `*=<password>` line to the password file (precedence exact → `*` → skip). If
   you do, keep `passwords.txt` at `chmod 600` and treat it as holding every
   un-overridden key.
6. **Monitoring:** if you assert exact intra-slot firing offsets, update them
   (mainnet attestation 4000 → 3999 ms; 6 s 2000 → 1999 ms; `attestation_time`
   genesis+4 → +3). The 2/3 aggregation (8000 ms mainnet) and 7 s (2333 ms)
   values are unchanged.
7. **Builder/MEV on non-mainnet:** Holesky/Sepolia/Hoodi builder-registration
   signing now works correctly; no operator action beyond running the same
   coordinated build on both `rvc` and `rvc-signer`.
8. Restart `rvc-signer` (it migrates the slashing DB on first open), then `rvc`.

No config-file format change is required; the wildcard keystore password is
additive and existing `pubkey=password` files keep working.
