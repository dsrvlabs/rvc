# rvc v0.4.0 — Low-findings closeout + code-quality pass

**Release date:** 2026-05-15
**Target tag:** `v0.4.0` on the same commit for `bin/rvc` and `bin/rvc-signer`.

This release closes all 11 Low audit findings (L-1..L-11), wires the
metrics-bind gate that was explicitly deferred from v0.3.0, performs the
post-GA code-quality consolidation (CQ-1..CQ-3), and accelerates the
test suite. Both binaries must be upgraded together.

---

## Breaking changes for operators

### L-10: Non-loopback metrics binds now refused (ISSUE-4.10)

**Action required for any operator binding the metrics endpoint to a
non-loopback address (e.g. `0.0.0.0:9000`, host LAN IP).**

The v0.3.0 notes flagged that the metrics-bind gate was scoped for a
later release and not wired. It is wired now. `rvc` reuses
`InsecureGate` (which defaults to `Refuse` since v0.3.0 / NFR-10) on the
non-loopback branch: startup hard-fails unless
`RVC_METRICS_ALLOW_NON_LOOPBACK=true` is set. Loopback binds remain the
secure default and pass silently.

Recommended fix: keep the metrics bind on loopback and front it with a
scraper (Prometheus, vmagent) that runs on the same host, or set the
env var if your environment isolates the metrics port at the network
layer.

### L-6: Keystore hot-reload is now opt-in (ISSUE-4.6)

**Action required for any operator who relied on `rvc-signer`'s default
30 s keystore-directory reloader.**

Two changes in `rvc-signer`:

1. The reloader is no longer spawned by default. To enable it, pass
   `--enable-hot-reload` or set `signer.enable_hot_reload = true` in
   the TOML config. Without that opt-in, `--reload-interval` is
   ignored and the reloader never starts. This removes the
   "directory-writable ⇒ automatic key injection" vector from the
   default deployment.
2. Each reload pass enforces a strict permission check on
   `keystore_dir`: mode must equal exactly `0o700` **and** owner UID
   must match the signer process's effective UID. Otherwise the pass
   is skipped with a `warn!` log; the reloader keeps running so a
   subsequent operator-fix is picked up. Unix-only — Windows ACLs are
   out of scope.

Operators upgrading: if you want hot reload, opt in **and** `chmod 700`
+ `chown` the keystore directory to the signer's user.

### L-1: DVT TLS handshake now pins per-peer SNI (ISSUE-4.1)

**Action required for DVT-enabled deployments using the per-peer allow-list.**

`GrpcPeerRequester::connect` now sets `.domain_name(peer.sni_cn)` per
peer before dialling. Rustls will refuse certs not issued for that
hostname, closing the cross-peer cert-substitution gap.

Operators must ensure their DVT allow-list TOML supplies `peer_cn` (and
optionally `addr`) values that match the SAN/CN of each peer's
certificate. A mismatch fails the handshake.

### L-9: Remote-key import re-resolves hostnames (ISSUE-4.9)

`POST /eth/v1/remote_keys` now re-resolves the URL's hostname at
import time via `tokio::net::lookup_host` and validates every resolved
IP against the existing private/reserved deny-list. A single denied
resolved IP refuses the import (DNS-rebinding mitigation).

No operator action required for URLs using public hostnames or IP
literals. Operators relying on a hostname that resolves to a
private/loopback/CGNAT address will now be refused at import time
where they previously passed startup-time validation.

### L-7: Per-token decrypt rate-limit on `import_keystores` (ISSUE-4.7)

`POST /eth/v1/keystores` now rate-limits the per-bearer-token decrypt
path. High-frequency repeated calls from a single token return HTTP 429.
This is a side-channel hardening; normal import flows are unaffected.

## Low findings closed (L-1..L-11)

### L-1 — Per-peer SNI pinning in DVT TLS handshake (ISSUE-4.1)

See "Breaking changes" above. `dvt/peer_client.rs` introduces
`PeerConnectInfo { addr, sni_cn }` and `dvt/allow_list.rs` adds the
optional `addr` field. A real-mTLS regression test
(`bin/rvc-signer/tests/dvt_sni_pinning_l1.rs`) spins up rcgen-issued
certs and asserts both the mismatch-refused and match-accepted paths.

### L-2 — Constant-time bearer-token compare without length leak (ISSUE-4.2)

`subtle::ConstantTimeEq for [u8]` short-circuits on length mismatch.
Wall-time was measurably different across length classes, leaking the
expected-token length. `ct_token_match` now pads both inputs to the
larger length, runs `ct_eq` on equal-length buffers, and ANDs the
result with a constant-time length-equality check. Folded into
`bearer_auth`. Includes the structural assertion that no length-mismatch
fast path remains. Polished in `8d94b8f` and `6959ad6`.

### L-3 — Canonical blob-commitment binding in `BlockAndBlobs` (ISSUE-4.3)

`eth-types` now binds KZG commitments canonically into the
`BlockAndBlobs` scope so commitment substitution at the wire is no
longer possible. Hardened in the review fix-up (`1f46e4d`).

### L-4 — `make_aggregation_bits` returns `None` for OOB committee index (ISSUE-4.4)

The previous code silently produced a malformed bitfield when given a
committee index outside the active set. It now returns `None`, and
callers propagate the rejection rather than emitting bad data.

### L-5 — Orchestrator regression guard against the `"head"` literal (ISSUE-4.5)

A `tests/` guard now fails the build if a `"head"` string literal
reappears anywhere in the orchestrator, preventing regression of the
slot-target bug that L-5 surfaced.

### L-6 — Keystore hot-reload opt-in + strict dir-mode check (ISSUE-4.6)

See "Breaking changes" above. New flag `--enable-hot-reload` and TOML
key `signer.enable_hot_reload`, plus a per-pass `check_keystore_dir_perms`
helper that requires `0o700` and a matching owner UID. `libc` added as
a Unix-only dependency for `geteuid()`. Truth-table covered by
`bin/rvc-signer/tests/hot_reload_l6.rs`.

### L-7 — Per-token rate limit on `import_keystores` decrypt (ISSUE-4.7)

Side-channel mitigation. See "Breaking changes" above.

### L-8 — `chmod 0o600` on slashing-db WAL/SHM sidecars (ISSUE-4.8)

SQLite WAL/SHM sidecar files inherit the open-mode of the main DB file
in some `rusqlite` builds. `crates/slashing` now explicitly `chmod`s
them to `0o600` on first open so a permissive umask doesn't leak
slashing-protection state to other local users.

### L-9 — Hostname re-resolution at remote-key import (ISSUE-4.9)

See "Breaking changes" above. New
`validate_remote_signer_url_runtime(url, allow_insecure)` async
variant; wired into `handlers.rs` at the single non-test call site.
**Known follow-up:** the same check is not yet applied at
`GrpcRemoteSigner::connect` / per-RPC reconnect — tracked for a Phase 5
release.

### L-10 — Non-loopback metrics-bind refusal (ISSUE-4.10)

See "Breaking changes" above. Reuses the `InsecureGate` from ISSUE-2.10 / A10
with `with_predicate`, since the production code only invokes the gate
in the non-loopback branch.

### L-11 — Test alignment for the L-9 URL change (ISSUE-4.11)

`test_keymanager_server_import_remote_key_lifecycle` updated to a
public IP literal (`8.8.8.8`) so the test reaches the inner key-import
path without depending on a CI DNS resolver. The hostname-resolution
branch is covered independently in `url_validator::tests::test_runtime_*`.

## Code-quality consolidations (CQ-1..CQ-3)

### CQ-1 — Crypto helpers

- `crypto::pubkey::CanonicalPubkey` (`f73db32`): single canonical
  representation for 48-byte BLS pubkeys with infallible-from-bytes /
  fallible-from-hex constructors.
- `crypto::hex::strip_prefix_strict` (`1710582`): tolerant-but-strict
  `0x` prefix stripping that rejects mixed-case `0X` and bare-hex
  inputs that previously slipped through ad-hoc trims.

### CQ-2 — Adoption + voluntary-exit discard fix

- Voluntary-exit signing now discards the slashing stage on signer
  error (`398df7c`), matching the M-1 stage/commit contract for
  attestations and blocks. RED test in `e03d869`.
- Parity tests (`7d14fa0`) prove `CanonicalPubkey` returns the same
  bytes as the prior ad-hoc parsers in `crates/slashing` and the
  orchestrator.
- `CanonicalPubkey` adopted across `crates/slashing` and the
  orchestrator (`8c2dc63`); `strip_prefix_strict` adopted at all four
  hex call sites in `crates/crypto` and `crates/rvc` (`fa1234c`).

### CQ-3 — `block-service::propose_block` consolidation

- RED test (`36a92c6`) showed the unvalidated `propose_block` accepted
  mismatched `proposer_index`.
- The unvalidated path was deleted and `propose_block_validated`
  renamed to `propose_block` (`04cb0be`). There is now exactly one
  proposer-validating entry point.
- The order-asserting test was replaced with structure that encodes
  the validation order in the code itself (`9021015`).

## Other fixes

### `validator-store::save_config` serialization (`1a7848e`)

`save_config` is now serialized with an internal mutex so two near-
simultaneous keymanager-API writes can't clobber each other with a
stale in-memory snapshot. Previously a slow write + a fast write
could leave the on-disk config reflecting the slow one.

### Phase-3 review fix-up: test-isolation env-var mutexes (`ec74f5c`)

`rvc-signer`, `grpc-signer`, and `crypto` tests that exercise the GA
`Refuse` contract (ISSUE-3.13) share a `std::sync::Mutex` to serialize
env-var mutation across the test binary, and unpoison the mutex on
prior-test panic so a single failure doesn't cascade.

### `test_audit_log_emitted_on_sign_request` cross-test stability

`integration_polish::tests::test_audit_log_emitted_on_sign_request`
previously used `tracing::subscriber::set_default` + a Registry layer;
the capture buffer was empty whenever a sibling test in the same lib
test binary primed the tracing callsite-interest cache before this
test ran. Migrated to the standard `tracing-test` crate
(`#[tracing_test::traced_test]` + `logs_contain(...)`), which installs
a process-global subscriber once and scopes the capture per test via
the test's own span. `tracing-test` added as a dev-dep on `rvc-signer`.

## Test-suite acceleration

The pre-acceleration baseline was captured in `637d0ce`. Net effect:
roughly `cargo nextest run --workspace` finishes substantially faster
end-to-end on a clean target.

- `crypto::EncryptionKdf::{Scrypt,Pbkdf2}With` + cheap-for-tests
  constructors (`704284c`, `3eb229c`, `015ca9b`): keystore fixtures
  across the workspace now use weak KDF params, dropping per-test
  keystore creation from hundreds of ms to single-digit ms.
- `bin/rvc-bin` integration tests hoist `build_binary()` behind
  `std::sync::Once` (`c833ebe`) so the release-mode binary is built
  exactly once per `cargo test` invocation.
- Wiremock-driven duty / reorg timeout tests use a `fast_timeouts()`
  helper (`a267521`) to shrink slot-spanning waits to test-scale.
- `make test-fast` (`9a0c54c`) prefers `cargo-nextest` when present
  and falls back to `cargo test --workspace` otherwise; coverage uses
  the same path.

## Upgrade path from v0.3.0

1. Stop `rvc` and `rvc-signer`.
2. Replace both binaries from the same `v0.4.0` build.
3. **If the metrics endpoint binds to a non-loopback address:** set
   `RVC_METRICS_ALLOW_NON_LOOPBACK=true` or rebind to loopback (L-10).
4. **If you depended on the default keystore hot-reload behaviour:**
   pass `--enable-hot-reload` (or set `signer.enable_hot_reload = true`)
   *and* `chmod 700` + `chown <signer-uid>` the keystore directory (L-6).
5. **If using DVT:** confirm each entry in the allow-list TOML has a
   `peer_cn` (and optionally `addr`) matching the peer certificate's
   SAN/CN (L-1).
6. **If using a hostname-based remote-signer URL:** confirm it resolves
   only to public addresses (L-9). IP literals are unaffected.
7. Restart `rvc-signer` then `rvc`.

No slashing-DB migration is required; the schema is unchanged from v0.3.0.
